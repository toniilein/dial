1. Overview

What is DIAL?

DIAL is a non-custodial namespace layer for Web3. A consumer registers a name like alice.dial; an enterprise registers its own corporate domain like .acme and issues names under it (finance.acme, treasury.acme). Each name maps to its owner's addresses on Canton and an EVM-compatible chain. Counterparties send to the name; they don't have to copy 42-character hex strings, and they get a verified identity attached to whatever they're sending to.

A Client (wallet, dapp, backend) calls DIAL. The Registry says who owns each namespace and which Resolver answers for it. The Resolver returns the actual records (chain addresses, text). The Registrar sells new namespaces. State is canonical at Pairpoint and propagated as signed copies to Canton + the EVM-compatible chain.

Core thesis. Web3 today asks users to share hex addresses with no identity behind them. DIAL replaces that with a non-custodial namespace layer — names like alice.dial or finance.acme resolve to addresses on every supported chain, and the name is backed by Vodafone Pairpoint identity attestation. Pairpoint-issued identity + the Vodafone brand are the moat.

2. Goals & Non-Goals

Goals (Phase 0)

  • Ship the DIAL App. The primary Phase 0 deliverable — the app where a client registers a namespace, maps it to chain addresses, and manages it.
  • Issue namespaces and corporate domains. Consumers and developers register names under DIAL's .dial domain; enterprises register their own corporate domain (e.g. .acme) and issue names under it.
  • Map namespaces to chain addresses on Canton and an EVM-compatible chain (DIAL's backend is source of truth; each chain holds a DIAL-signed copy).
  • Be non-custodial. Pairpoint operates the AA infrastructure; user signing keys never live in DIAL.
  • Attest identity for every persona — enterprise KYB, consumer KYC — via Vodafone Pairpoint Identity Hub. DIAL stores only the attestation hash.
  • Expose a Wallet SDK and resolver API so developers can integrate DIAL namespace lookup natively.

Non-Goals (Phase 0)

  • Not custodial. DIAL never holds user keys.
  • Not a marketplace. Secondary trading is deferred.
  • Not an agent / IoT platform. Agent Plane, bulk issuance, fleet management are deferred.
  • Not a cross-chain bridge. DIAL never moves value between chains.

3. Personas (Phase 0)

Who uses DIAL, and what each persona needs from it.

Enterprise Acme

A business that buys its own verifiable corporate domain — .acme — and issues names under it like finance.acme or treasury.acme, one per team, function, or service. Goes through Pairpoint Identity Hub verification once, maps each name to its Canton party and EVM address, then uses the API for resolution and ownership operations — every name rolling up to the verified .acme parent.

Developer Dev

Integrates the DIAL Wallet SDK or resolver API into a wallet, dapp, or backend so end-users can resolve a name like alice.dial or finance.acme to a Canton or EVM-compatible address. One-line resolver call + verifiable signed attestations.

Consumer Alice

An individual who wants a personal namespace like alice.dial that maps to her own Canton and EVM-compatible addresses, so people can pay her or find her with a readable name instead of a raw 0x… address. Verifies once through Pairpoint Identity Hub, holds the namespace non-custodially, and uses it to receive funds and prove ownership.

6. Solution Architecture

Phase 0 MVP — namespace issuance for developers, enterprises, and consumers. Enterprises register their own verifiable corporate domain (e.g. .acme) and issue names under it; developers and consumers register names under DIAL's .dial domain. Clients: Wallet SDK, Resolver API, and the DIAL App. Non-custodial — Pairpoint operates account-abstraction infrastructure (bundler / paymaster / sponsored gas); user keys never live in DIAL. Vodafone Pairpoint Identity Hub as the identity gate for every persona. Built on Canton + one EVM-compatible chain from day one.

Out of Phase 0 scope and deferred to later phases (§12 Roadmap): marketplace, Agent Plane, operator Console, browser interface, third-party Wallet SDK, additional EVM and non-EVM chains, additional Identity Hub capabilities.

What each block does

① Clients

Phase 0 = developers, enterprises, and consumers. The Wallet SDK embeds resolution into wallets and dapps; the Resolver API is the public REST/gRPC endpoint; the DIAL App is the first-party interface for registering and managing names.

② DIAL Platform

Mirrors the ENS three-component pattern. Three cores + two supporting services:

  • Registry — ownership. For every domain and namespace: who owns it, which Resolver answers, expiry.
  • Resolver — records. The actual data: chain addresses, text records, contenthash.
  • Registrar — issuance. Issues corporate domains to enterprises and names under .dial to developers and consumers; owners create subnames under their own domain (register, renew, available).
  • Billing — USDC payment collection for primary issuance + annual renewal.
  • Chain Sync — propagates DIAL-signed Registry + Resolver state to both on-chain copies.

③ Vodafone Pairpoint Identity Hub

Phase 0 identity gate. DIAL holds no PII, just the attestation hash returned by Pairpoint.

④ On-chain Registries

Each chain hosts a DIAL-signed DialRegistry + Resolver contract pair, mirroring the ENS pattern. Canton uses Daml contracts; the EVM-compatible chain uses Solidity (DialRegistry.sol, DialResolver.sol). The DIAL Backend is canonical; on-chain copies let any wallet or dapp verify locally on whichever chain it lives on.

6.3 Data Tier

The DIAL backend (Postgres + Vault) is the source of truth. Each supported chain holds a signed copy in its DialRegistry contract — useful for on-chain verification by wallets and dapps on that chain, but not authoritative. Critically, PII never lives in DIAL — it stays with Vodafone IDH; DIAL only holds the resulting attestation hash.

6.4 Event-Driven Backbone

Inter-service communication uses an event bus (Kafka or NATS JetStream) for asynchronous flows. Synchronous reads use gRPC; public APIs are REST + GraphQL.

Why event-driven? Chain commits are async with very different finality characteristics (Canton seconds, the EVM-compatible chain ~12s, future chains minutes). An event bus decouples user-facing latency from settlement finality across chains and gives us a uniform audit substrate.

6.6 How ENS Is Built (Reference)

DIAL's Registry / Resolver / Registrar split is borrowed from ENS — the Ethereum Name Service. This is the prior art: how ENS does it on-chain, so the choices in §6.1–§6.5 have context.

Everything in ENS is keyed by namehash (ENSIP-1) — names are never stored as strings on-chain. A name is first normalized (ENSIP-15), then namehash("alice.eth") recursively hashes each label into one fixed 32-byte node, and every contract below addresses names by that node.

How a name gets resolved

Resolution is a read: turn the name into a node, ask the Registry which resolver answers for it, then ask that resolver. A modern resolver can also answer off-chain via CCIP-Read (ERC-3668) — the client fetches from a gateway and the resolver verifies a proof.

7. Data Model

Two-level namespace: a domain is the apex (.dial operated by DIAL, or a corporate domain like .acme), a namespace is a name under a domain (alice.dial, finance.acme), and a subname is a third level under a namespace (pay.alice.dial). Records (chain bindings, text) attach to namespaces and subnames — never to the apex domain itself.

erDiagram OWNER ||--o{ DOMAIN : owns OWNER ||--o{ NAMESPACE : owns OWNER ||--o{ SUBNAME : owns OWNER ||--|| KYC_RECORD : "verified via" DOMAIN ||--o{ NAMESPACE : "parent of" NAMESPACE ||--o{ SUBNAME : "parent of" NAMESPACE ||--o{ CHAIN_BINDING : has NAMESPACE ||--o{ TEXT_RECORD : has SUBNAME ||--o{ CHAIN_BINDING : has SUBNAME ||--o{ TEXT_RECORD : has NAMESPACE ||--o{ BILLING_RECORD : "billed via" DOMAIN ||--o{ BILLING_RECORD : "billed via" OWNER { uuid id PK enum kind "CONSUMER or ENTERPRISE" string canton_party string evm_address uuid kyc_record_id FK timestamp created_at } KYC_RECORD { uuid id PK enum kind "KYC or KYB" string provider "Pairpoint IDH" string attestation_hash "no PII stored" timestamp verified_at timestamp expires_at } DOMAIN { uuid id PK string label "dial or acme" uuid owner_id FK enum issuance_policy "OPEN, CURATED, or CLOSED" enum status "ACTIVE, GRACE, or EXPIRED" timestamp registered_at timestamp expires_at } NAMESPACE { uuid id PK string fqn "alice.dial or finance.acme" uuid domain_id FK uuid owner_id FK string resolver "default or custom URI" enum status "ACTIVE, GRACE, or EXPIRED" timestamp registered_at timestamp expires_at } SUBNAME { uuid id PK string fqn "pay.alice.dial" uuid namespace_id FK uuid owner_id FK enum status "ACTIVE or RELEASED" timestamp created_at } CHAIN_BINDING { uuid id PK uuid target_id FK "namespace or subname" string chain_id "CAIP-2 e.g. canton:omnibus, eip155:<chain>" string address "CAIP-10 account" string proof_signature "EIP-191/712 or Daml party auth" timestamp bound_at timestamp revoked_at } TEXT_RECORD { uuid id PK uuid target_id FK string key "e.g. avatar, email, url, lei" string value timestamp updated_at } BILLING_RECORD { uuid id PK uuid target_id FK enum sku "NAME or DOMAIN_PREMIUM or DOMAIN_STANDARD or DOMAIN_BASIC" decimal list_usdc decimal paid_usdc decimal discount_pct enum kind "REGISTRATION or RENEWAL" timestamp captured_at string receipt_ref }

State distribution (Phase 0):

  • DIAL backend (canonical): domains, namespaces, subnames, chain bindings, text records, billing — Postgres. Vault / HSM holds DIAL's own signing key (not user keys — DIAL is non-custodial).
  • Canton + EVM-compatible chain: each hosts a DialRegistry contract storing hash(fqn, owner, expires) + DIAL signature + lastUpdatedAt — a signed mirror of the canonical row.
  • Vodafone Pairpoint IDH (external): PII and identity documents — KYC docs, KYB business records, liveness samples. DIAL only holds the resulting attestation hash.
  • Redis: resolver hot cache (fqn → chain-keyed address record), session state, rate-limit counters, cart contents.

8. Key User Flows

End-to-end paths a person walks through in the DIAL App, mapped to the services that handle each step. Every flow terminates in a state visible at the public Resolver (or, for sign-in, in an authenticated session). Solid arrows are synchronous; dotted arrows are events on the bus.

8.1 Sign in to the DIAL App

OAuth identity → Pairpoint AA derives the on-chain account. Keys never leave the user's device or live in DIAL.

sequenceDiagram participant U as User participant APP as DIAL App participant OIDC as Google / Apple OIDC participant AA as Pairpoint AA participant GW as DIAL API U->>APP: tap Sign in APP->>OIDC: OAuth flow OIDC-->>APP: id_token APP->>AA: derive account from id_token AA-->>APP: Canton party + EVM address APP->>GW: open session (id_token + addresses) GW-->>APP: session · no identity yet

What DIAL stores at this point: session ↔ derived addresses. No PII; no IDH attestation yet — that lands at registration (§8.3).

8.2 Search & price a name

Live availability check with a USDC quote — runs as the user types. Sub-300ms target.

sequenceDiagram participant U as User participant APP as DIAL App participant GW as DIAL API participant REG as Registrar participant BI as Billing U->>APP: type "alice" APP->>APP: normalise (lowercase, IDNA, ≤63 chars) APP->>GW: GET /v1/registrar/available?name=alice.dial GW->>REG: validity + reserved + active checks REG-->>GW: available GW->>BI: GET /v1/billing/quote BI-->>GW: 240 USDC · 180 USDC verified GW-->>APP: { available, price, listPrice, discount }

Discount surfaces here. If the user is signed in and KYC-verified, Billing returns the verified-tier quote (Phase 0: 25% off) — the cart shows it before Pay (§4.7 row 7.5).

8.3 Register a name (consumer / developer)

Cart → sign-in → identity verification (KYC) → pay → on-chain mirrors. The cart survives sign-in / sign-out so users can fill a basket before logging in.

sequenceDiagram participant U as User participant APP as DIAL App participant GW as DIAL API participant IDH as Pairpoint IDH participant BI as Billing participant REG as Registrar participant REGY as Registry participant SYNC as Chain Sync U->>APP: add alice.dial to cart U->>APP: open checkout APP-->>U: Step 1 — Review alt not signed in APP-->>U: Step 2 — Sign in (§8.1) end alt not KYC-verified APP-->>U: Step 3 — Verify identity APP->>IDH: KYC (personal · document · liveness) IDH-->>APP: attestation_hash end APP-->>U: Step 4 — Pay APP->>BI: capture USDC BI-->>APP: paid APP->>GW: POST /v1/registrar/register GW->>REG: register(alice.dial, owner, 1yr, attestation_hash) REG->>REGY: write ownership row REGY-)SYNC: identity.issued SYNC-)SYNC: signed write → Canton + EVM REG-->>APP: ACTIVE APP-->>U: Step 5 — Done · open name

The IDH step runs once per persona. Subsequent registrations for the same signed-in user skip steps 2–3.

8.4 Register a corporate domain (enterprise)

KYB-gated entry point for an enterprise. After registration, the enterprise issues names under the new domain (§8.5).

sequenceDiagram participant U as Admin participant APP as DIAL App participant IDH as Pairpoint IDH (KYB) participant BI as Billing participant REG as Registrar participant REGY as Registry participant SYNC as Chain Sync U->>APP: search ".acme" in domain mode APP->>REG: availability + length-tier price REG-->>APP: available · 2,400 USDC/yr U->>APP: add to cart · checkout alt not KYB-verified APP->>IDH: KYB (business · corporate · beneficial owners) IDH-->>APP: business_attestation_hash end APP->>BI: capture USDC (domain SKU) BI-->>APP: paid APP->>REG: register(.acme, owner, 1yr, business_attestation_hash) REG->>REGY: write apex domain row REGY-)SYNC: identity.issued (domain) APP-->>U: .acme issued · Issue names →

Apex doesn't bind a chain. The domain is a container — only the names under it carry chain addresses (§4.1 row 1.5).

8.5 Issue a name under your corporate domain

Domain owner creates names like finance.acme. No second KYB step — the parent domain's attestation covers the children. No additional billing cycle by default — issuance under an owned domain is included.

sequenceDiagram participant U as Admin participant APP as DIAL App participant REG as Registrar participant REGY as Registry participant SYNC as Chain Sync U->>APP: open .acme · Issue name U->>APP: label = "finance" APP->>REG: availability under .acme REG-->>APP: available U->>APP: confirm APP->>REG: register(finance.acme, parent=.acme, owner) REG->>REGY: write namespace row REGY-)SYNC: identity.issued SYNC-)SYNC: signed write → Canton + EVM APP-->>U: finance.acme issued

8.6 Bind a chain address (manage records)

Owner adds or updates a Canton party / EVM address on a name they own. Proof-of-control is required per chain — DIAL never trusts a self-declared address.

sequenceDiagram participant U as Owner participant APP as DIAL App participant AA as Pairpoint AA participant GW as DIAL API participant RES as Resolver participant SYNC as Chain Sync U->>APP: open finance.acme · Records U->>APP: enter EVM address 0x… APP->>AA: request EIP-712 proof-of-control AA-->>APP: signature APP->>GW: POST /v1/resolver/finance.acme/addr/evm GW->>RES: verify signature · write record RES-)SYNC: chain.binding.added SYNC-)SYNC: signed write → EVM DialRegistry RES-->>APP: 200 · current records APP-->>U: saved · visible to public Resolver

Why proof-of-control? Prevents an owner binding an address they don't actually control. EIP-191/712 for EVM, Daml party auth for Canton (§4.5 row 5.4).

8.7 Public resolution (read path)

The fast path. A wallet, dapp, or SDK asks for a name; Resolver answers from cache with a DIAL-signed attestation the caller can verify locally.

sequenceDiagram participant C as Caller (SDK / dapp) participant GW as DIAL API participant RES as Resolver participant RD as Redis participant PG as Postgres C->>GW: GET /v1/resolver/alice.dial GW->>RES: lookup RES->>RD: cache hit? alt cache hit RD-->>RES: cached record else miss RES->>PG: read namespace + bindings PG-->>RES: record RES->>RD: cache (TTL) end RES-->>GW: addresses + DIAL-signed attestation GW-->>C: JSON · sub-300ms Note over C: SDK verifies attestation signature client-side

8.8 Renew & Release

Two lifecycle paths — both bypass the cart. Renewal extends expiry; release sends the name to a 30-day grace period before it returns to the available pool.

sequenceDiagram participant U as Owner participant APP as DIAL App participant BI as Billing participant REGY as Registry participant SYNC as Chain Sync rect rgba(61,220,151,0.06) Note over U,SYNC: Renew U->>APP: open name · Renew APP->>BI: capture USDC (renewal) BI-->>APP: paid APP->>REGY: extend expires_at +1yr REGY-)SYNC: identity.renewed end rect rgba(255,180,84,0.06) Note over U,SYNC: Release U->>APP: open name · Release · confirm APP->>REGY: mark GRACE (30 days) REGY-)SYNC: identity.revoked Note over REGY: after grace → EXPIRED → available pool end

Grace period. Released names sit in GRACE for 30 days before returning to the available pool — protects against accidental release and gives the owner a window to renew.

10. Security & Compliance

10.1 Threat Model (Phase 0)

ThreatMitigation
Name squatting / phishing (homoglyphs)IDNA2008 + custom skeleton-string check; reserved-name list.
Account takeoverDIAL is non-custodial — user keys live on device, so a server breach cannot drain accounts. Step-up Pairpoint IDH re-verification on sensitive ops. Biometric 2FA + device binding in the DIAL App.
Resolver poisoningResolver returns DIAL-signed attestations; SDK verifies signature client-side and pins DIAL's published public key.
Insider key misuse (DIAL signing key)DIAL's own signing key in Vault / HSM with dual-control and audit log. User keys not held by DIAL — non-custodial.
Cross-chain replayEvery signed attestation carries domain separators (chainId + contract address); sync writes are nonce-protected per chain.
Stale on-chain copyDialRegistry on Canton and EVM-compatible chain expose lastUpdatedAt; SDKs refuse to verify against attestations older than N hours.
Bridge / cross-chain value theftOut of scope by design — DIAL never bridges value. Each chain settles natively.

10.2 Regulatory Posture (Phase 0)

  • GDPR — PII stays in Vodafone Pairpoint IDH; DIAL holds only attestation hashes.
  • MiCA — DIAL is non-custodial. Identity issuance is the product; settlement happens in user-controlled wallets — DIAL doesn't custody crypto-assets.
  • AML / CTF — Sanctions screening at Pairpoint IDH KYC.
  • SOC 2 Type II targeted within 12 months of GA.
